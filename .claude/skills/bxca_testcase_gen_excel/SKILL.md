주어진 디렉토리 `$ARGUMENTS` 아래의 소스 파일을 분석하여 테스트케이스를 Excel(.xlsx) 파일로 생성합니다.

## 실행 절차

### Step 0: 경로 유효성 검증
1. `$ARGUMENTS` 경로가 존재하는지 확인합니다.
2. 경로가 존재하지 않거나 디렉토리가 아닌 경우 사용자에게 알리고 중단합니다.
3. 해당 경로 아래에 분석 가능한 소스 파일이 1개 이상 있는지 확인합니다.
4. 소스 파일이 0개이면 "분석 대상 소스 파일이 없습니다"라고 알리고 중단합니다.

### Step 1: 대상 파일 수집
1. `$ARGUMENTS` 경로 아래의 소스 파일을 Glob으로 수집합니다.
   - 지원 확장자: `.java`, `.py`, `.js`, `.ts`, `.jsx`, `.tsx`, `.go`, `.cs`, `.kt`, `.rb`, `.php`, `.c`, `.cpp`, `.rs`
   - `node_modules/`, `build/`, `dist/`, `target/`, `.gradle/`, `__pycache__/`, `.git/` 등 빌드/의존성 디렉토리는 제외합니다.
   - 테스트 파일(`*Test.java`, `*_test.py`, `*.test.ts`, `*.spec.ts` 등)은 제외합니다.
   - 분석 제외 대상: `main()` 메서드만 있는 진입점 클래스, `@Configuration` 설정 클래스, 인터페이스(Mapper 등), 상수 enum, 단순 getter/setter 엔티티
2. 수집된 파일 목록을 사용자에게 간략히 보여줍니다.

### Step 2: 소스 코드 분석 및 테스트케이스 도출
각 소스 파일에 대해 Read 도구로 코드를 읽고 아래 기준으로 테스트케이스를 도출합니다.
분석 순서는 **Controller → Service → Config/기타** (레이어 순서)로 진행합니다.

**분석 대상:**
- **분기문(if/else, switch/case, 삼항연산자)**: 각 분기 조건별로 테스트케이스 생성
- **반복문의 경계 조건**: 빈 컬렉션, 단일 요소, 다수 요소
- **예외 처리(try/catch/throw)**: 정상 경로와 예외 경로 각각
- **입력 검증 로직**: null 체크, 범위 검사, 형식 검증 등
- **API 엔드포인트**: 정상 요청, 잘못된 요청, 인증/권한 검사
- **비즈니스 로직**: 계산, 상태 변경, 데이터 변환 등 핵심 로직
- **주석/JavaDoc**: `@param`, `@throws`, `TODO`, `FIXME` 등에서 테스트 힌트 추출

**테스트케이스 필드 작성 규칙:**

| 필드 | 작성 규칙 |
|------|-----------|
| **테스트케이스ID(소스파일명)** | 해당 테스트케이스를 추출한 소스 파일명 (예: `EmpService.java`) |
| **테스트케이스명** | 테스트 목적을 간결하게 기술 (예: `직원 삭제 시 존재하지 않는 ID 처리`) |
| **테스트내용** | 구체적인 테스트 시나리오 설명 (예: `존재하지 않는 직원 ID로 삭제 요청 시 적절한 예외가 발생하는지 확인`) |
| **테스트조건** | 실제 코드의 분기 조건 + 관련 주석 포함 (예: `emp == null // 직원이 존재하지 않는 경우`) |
| **예상결과** | 해당 조건에서의 기대 동작 (예: `ResourceNotFoundException 발생 또는 404 응답 반환`) |
| **중요도** | 테스트 우선순위: **상** / **중** / **하** |
| **테스트결과** | 항상 공란으로 둡니다 |

**중요도 판단 기준:**
- **상**: CRUD의 CUD 핵심 경로, 보안 취약점(SQL Injection 등), 트랜잭션 롤백, 입력 검증 실패, 인증/권한
- **중**: 조회 시 존재/미존재 분기, 예외 발생 경로, insert/delete 실패 경로
- **하**: 전체 목록 단순 조회, 빈 목록 반환, 로깅 확인, 설정 초기화

### Step 3: Excel 파일 생성

1. 도출된 테스트케이스를 JSON 형식으로 임시 파일에 저장합니다.
2. 이 스킬의 보조 스크립트 `bxca_testcase_gen_excel.py`를 실행하여 Excel 파일을 생성합니다.
3. **생성 완료 후 임시 JSON 파일을 반드시 삭제합니다.**

**출력 파일 규칙:**
- 파일명: `{디렉토리이름}_테스트케이스.xlsx` (예: `spapp_테스트케이스.xlsx`)
- 디렉토리 이름은 `$ARGUMENTS` 경로의 마지막 디렉토리명을 사용
- 파일 저장 위치: `$ARGUMENTS` 경로 (분석 대상 디렉토리의 루트)
- **동일 파일이 이미 존재하면** `_v2`, `_v3` ... 순서로 버전 suffix를 붙여 생성합니다

**실행 명령:**
```bash
# JSON 임시 파일 생성 후
python "<this_skill_directory>/bxca_testcase_gen_excel.py" "<output_path>" "<temp_json_path>"

# 임시 JSON 파일 삭제
rm "<temp_json_path>"
```

여기서 `<this_skill_directory>`는 이 스킬의 실제 디렉토리 경로입니다.
예: `c:\DevHome\repository\customskills\.claude\skills\bxca_testcase_gen`

**JSON 형식:**
```json
{
  "sheet_name": "테스트케이스_자동생성",
  "target_directory": "c:\\DevHome\\repository\\spapp",
  "test_cases": [
    {
      "id": "EmpService.java",
      "name": "직원 생성 - 정상 입력",
      "content": "유효한 직원 정보로 생성 요청 시 정상 등록되는지 확인",
      "condition": "emp != null && emp.getName() != null",
      "expected": "직원 정보가 DB에 저장되고 생성된 직원 객체 반환",
      "priority": "상"
    }
  ]
}
```

### Step 4: 결과 보고
- 생성된 Excel 파일 경로를 사용자에게 알려줍니다.
- 총 분석 파일 수와 도출된 테스트케이스 수를 요약합니다.
- 파일별 테스트케이스 수를 간략히 표로 보여줍니다.

## Excel 시트 구성

### 시트 1: 요약 (첫 번째 시트)
- 맨 위에 생성 일시, 작업 대상 디렉토리 표시
- 파일별 테스트케이스 건수 집계 테이블
- 중요도별(상/중/하) 건수 집계

### 시트 2: 테스트케이스_자동생성
- No, 테스트케이스ID(소스파일명), 테스트케이스명, 테스트내용, 테스트조건, 예상결과, 중요도, 테스트결과

## Excel 서식 규칙

### 전문적 폰트
- 모든 셀에 일관된 전문적 폰트 사용 (Arial)

### 스타일 표준
- **헤더**: 파란 배경(#4472C4), 흰색 볼드 텍스트, 가운데 정렬, wrap_text
- **데이터 셀**: 테두리(thin), 위쪽 정렬, wrap_text
- **No / 중요도 / 테스트결과 열**: 가운데 정렬
- **자동 필터**: 테스트케이스 시트 전체에 적용
- **틀 고정**: 헤더 행 아래 (A2)
- **중요도 색상**: 상=연한 빨강 배경, 중=연한 노랑 배경, 하=배경 없음

### 파일 무결성
- 수식 사용 시 반드시 재계산하여 수식 에러(#REF!, #DIV/0! 등) 없이 전달

## 주의사항
- 소스 파일이 너무 크면(500줄 이상) public 메서드 위주로 분석합니다.
- 단순 getter/setter, 설정 클래스, 상수 정의 파일 등은 테스트케이스 도출 대상에서 제외합니다.
- 한국어로 테스트케이스를 작성합니다.
- openpyxl이 미설치 상태라면 스크립트가 자동으로 pip install합니다.
